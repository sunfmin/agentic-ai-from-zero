# 第 15 章 —— 把代码深化

_上次校对：2026-06，对齐 Claude Code v2.x_

前四章是给 **新功能** 做规划。这一章换一个入口：对着 **已有代码** 跑，找出架构上的毛病并修。用的 skill 是 `/improve-codebase-architecture`。

它要找的东西有个专门说法：**深化机会**（deepening opportunity）—— 把一个「浅模块」改造成「深模块」。AI 写代码很快，也就同样很快把代码库堆成一个 **烂泥球**。作者建议每隔几天就对自己的代码库跑一次这个 skill，把熵增按住。

## 先学几个词：深模块 vs 浅模块

这个 skill 通篇用一套精确的词汇说话，先把核心两个弄懂。

- **module（模块）** —— 任何「有接口、有实现」的东西：一个函数、一个类、一个包都算。
- **interface（接口）** —— 用对这个模块 **必须知道的一切**：不只是函数签名，还包括它的前提、顺序要求、出错方式、性能特征。
- **deep（深）/ shallow（浅）** —— 一个模块 **深**，是指「**小接口背后藏着大量行为**」：你只要学一点点（小接口），就能用上很多（大实现）。一个模块 **浅**，是指「接口几乎和实现一样复杂」—— 你为了用它要学的，跟它自己干的事差不多多。**浅模块是要避免的。**

```
深模块（要的）              浅模块（避免）
┌──────────────┐          ┌────────────────────────┐
│   小接口     │          │       大接口           │
├──────────────┤          ├────────────────────────┤
│              │          │  薄实现（基本只是转发） │
│   大实现     │          └────────────────────────┘
│  （藏起复杂）│
└──────────────┘
```

判断一个模块是不是「白占地方」，用 **删除测试**：假想把它删掉。如果复杂度跟着 **消失** 了，说明它本来就是个转发壳子（浅）；如果复杂度 **散到了一堆调用方身上**，说明它在挣自己的饭钱（深）。

再记两个词，报告里会反复出现：

- **leverage（杠杆）** —— 调用方得到的好处：学一点接口，换来很多能力。一份实现，养活 N 个调用点。
- **locality（局部性）** —— 维护者得到的好处：改动、bug、知识都 **集中在一处**，而不是散在各个调用方。改一次，处处都对。

> 💡 还有两个会出现的词。**seam（缝）**：能在不改它本身的前提下替换其行为的位置（也就是测试挂钩的地方，上一章见过）。**adapter（适配器）**：在缝上满足接口的具体实现。一条经验：「**一个 adapter 只是个假想的缝，两个 adapter 才是真的缝**」—— 没有两种东西真的在那儿替换，就别硬开一条缝。

## 动手：找一个代码库来跑

> 🛠 试一试
>
> **首选**：如果你已经动手写了点 reading-list 的代码，或者手头有任何一个小代码库 —— 用它，看 skill 描述 **你自己** 写的代码最过瘾。
>
> **没有的话**：本书附带了一个极小的示例代码仓库 `tutorial/sample-project/`（一段故意写得有点拧巴的「下单」流程）。把本书这个仓库 clone 到本地，`cd` 进那个文件夹：
>
> > ```
> > cd path/to/this-tutorial/tutorial/sample-project
> > claude
> > ```
>
> 你应该看到：Claude Code 在那个小项目里启动。

> 💡 这个 skill 只 **读** 代码、出报告，**不执行** 任何东西。所以示例项目不需要能跑、不用装依赖。

跑它：

> 🛠 试一试
>
> 在 prompt 上输入：`/improve-codebase-architecture`
>
> 你应该看到：Claude 先（如果有的话）读 `CONTEXT.md` 和 ADR 拿术语，然后用一个 explore agent 把代码走一遍，**找哪儿让人别扭** —— 哪个模块浅、哪儿的逻辑漏过了缝、哪儿测起来别扭。（示例项目里，它大概会盯上「定价逻辑散在三个地方」「校验那层几乎是空壳」这类。）

## 报告：一个自带的 HTML 网页

走查完，它 **不会** 在终端里堆一长串文字。它写一个 **自包含的 HTML 文件** 到系统临时目录（不落进你的仓库），然后自动用浏览器打开，并告诉你文件的绝对路径。

> 🛠 试一试
>
> 等它写完。
>
> 你应该看到：浏览器弹出一个排版讲究的报告页（用了 Tailwind 做样式、Mermaid 画关系图）。终端里会给你一句类似「报告在 `/var/folders/.../architecture-review-<时间戳>.html`」。

报告里每个 **候选**（candidate）是一张卡片：

- **Files** —— 牵涉哪些文件/模块；
- **Problem** —— 现在的架构哪儿别扭（一句话）；
- **Solution** —— 改成什么样（一句话）；
- **Before / After 图** —— 并排画出「现在多浅」和「深化后多深」；
- **Wins** —— 用 leverage / locality 的话说清收益（比如「定价逻辑不再漏过缝」「测试只打一个接口」）；
- **推荐强度** badge —— `Strong`（强烈建议）/ `Worth exploring`（值得一看）/ `Speculative`（拍脑袋）。

最后还有一个 **Top recommendation** 段，告诉你「要动，先动哪个，为什么」。

> 💡 如果某个候选跟你的某条 ADR 冲突，报告 **不会** 偷偷推翻它，而是明确标出来（「这跟 ADR-0003 冲突 —— 但值得重开，因为……」），让你自己判断要不要重议那条 ADR。

读完报告，它会问你一句：**「这几个里，你想深入聊哪一个？」** —— 注意，到这一步它 **还没** 提出任何具体接口设计。挑一个。

## 选中之后：又是一轮逼问

你挑了一个候选，`/improve-codebase-architecture` 就切回你熟悉的 `/grilling` 模式（第 12 章那套），跟你一起把这个深化的设计走一遍：约束是什么、依赖是什么、深化后的模块长什么形状、缝后面藏什么、原来的测试哪些还活着。

> 🛠 试一试
>
> 挑「Top recommendation」那个，跟着它走几轮问答。
>
> 你应该看到：跟第 12 章一样的逼问 —— 一次一问、每问带推荐。而且 domain-modeling 的副作用又会出现：
>
> - 如果深化后的模块要起一个 `CONTEXT.md` 里 **还没有** 的名字，它会把这个词 **加进** `CONTEXT.md`；
> - 如果聊着聊着把某个模糊的词说清了，它当场更新 `CONTEXT.md`；
> - 如果你 **否决** 了这个候选、而且理由很「承重」（未来的人会因为不知道这个理由而重复提同样的建议），它会问你要不要记成一份 ADR，免得下次架构 review 又把它翻出来。

逼问走完，你手上就有了一个 **想清楚的深化方案** —— 接下来真正去改代码，可以交给 `/tdd` 之类的 skill（第 16 章给门），也可以自己动手。这个 skill 的职责到「把方案想清楚」为止。

## 小结

- `/improve-codebase-architecture` 是独立入口，面向 **已有代码**：找 **浅模块**，把它 **深化** 成深模块。
- 核心词汇：deep/shallow module、interface、seam、adapter、**leverage**（调用方的杠杆）、**locality**（维护者的局部性）、**删除测试**。
- 产物是一个 **自包含 HTML 报告**（写到系统 temp、不进仓库），每个候选有 before/after 图和推荐强度 badge。
- 选中候选后切回 `/grilling` 把设计走一遍；过程中照样会更新 `CONTEXT.md`、按需提议 ADR。
- 它只负责把深化方案 **想清楚**；真正改代码是下一步的事。
