# 第 13 章 —— 把决策综合成 PRD

_上次校对：2026-06，对齐 Claude Code v2.x_

上一章你被逼问了一轮，决策定了稿、留了痕（`CONTEXT.md` + ADR）。这一章把这些决策 **综合成一份 PRD**（product requirements document，产品需求文档），并发成一个 GitHub issue。用的 skill 是 `/to-prd`。

这一章最反直觉的一点：**`/to-prd` 不会问你任何问题。** 它的设计就是「不面试用户，直接综合已有的对话和决策」。原因下面会讲清楚。

## 它做什么

`/to-prd` 拿到「当前这场对话的上下文 + 仓库现状（`CONTEXT.md`、ADR、已有代码）」，产出一份结构固定的 PRD，发成一个带 `ready-for-agent` 标签的 issue。中间它只会停下来跟你确认 **一件** 事 —— 见下面的 seam。

> 💡 **最好在 grill 的同一个 session 里接着发 `/to-prd`**，上下文最全。但就算你关了再开一个新 session 也没事 —— 因为上一章的决策已经落在 `CONTEXT.md` 和 ADR 里了，`/to-prd` 会先读它们。这正是「当场留痕」的回报：决策不依赖某一场对话还开着。

## 唯一会停下来问你的：seam

PRD 写之前，`/to-prd` 会先勾一下「这个功能打算在哪些 **seam**（缝）上测」，并拿给你确认。

> 💡 **seam** 是个借来的词：代码里一个「你能在不改动它本身的前提下、替换掉它行为」的位置 —— 也就是你挂测试、塞假数据的地方。第 15 章会把这个词讲透。这里你只需知道：`/to-prd` 倾向于 **少开缝**。它的原则是「优先复用已有的缝、取尽量高的那条缝、缝越少越好 —— 理想是只有一条」。缝少，意味着测试集中、改动集中。

> 🛠 试一试
>
> 在仓库里（接着上一章的 session 最好），输入：`/to-prd`
>
> 你应该看到：Claude 先探索一下仓库、读 `CONTEXT.md` 和 ADR，然后给你看它打算在 **哪条 seam** 上测这个功能，问你「这跟你的预期对得上吗」。看一眼答「对」或者纠正它。

## PRD 长什么样

确认完 seam，Claude 直接写 PRD，按一个固定模板。七节：

| 节 | 装什么 |
|---|---|
| **Problem Statement** | 用户视角的问题 |
| **Solution** | 用户视角的解法 |
| **User Stories** | 一长串「作为<谁>，我想要<什么>，以便<好处>」，要 **穷尽**，覆盖功能各个面 |
| **Implementation Decisions** | 要建/改哪些模块、它们的接口、架构决定、schema、API 约定 —— 但 **不写具体文件路径和代码** |
| **Testing Decisions** | 什么算好测试（只测外部行为）、测哪些模块、有没有可参照的现有测试 |
| **Out of Scope** | 明确 **不** 做什么 —— 反向定义，减少未来扯皮 |
| **Further Notes** | 其它备注 |

> 💡 注意它 **不写文件路径和代码片段** —— 这些过期太快。唯一例外：如果之前的探索产出了某个比文字更精确的小片段（一个状态机、一份 schema、一个类型），可以内嵌，但只留「承载决策」的那几行，不是能跑的 demo。

写完，它把这份 PRD publish 成一个 GitHub issue，贴上 `ready-for-agent` 标签 —— 意思是「这份东西已经具体到，一个不在场的 agent 可以直接拿去开工」。

> 🛠 试一试
>
> 让它继续（你不用再答什么）。
>
> 你应该看到：Claude 调 `gh issue create` 发出一个 issue，回给你一个链接（类似 `https://github.com/<你>/<repo>/issues/1`）。打开看看 —— 七节俱全，user stories 一长串，每条都用上了 `CONTEXT.md` 里统一过的术语。

> 💡 **小细节：它会自己补标签。** 上一章 `/setup` 只是把「用哪五个 triage 标签」 **写进了** `docs/agents/triage-labels.md`，GitHub 仓库本身还没真的创建这些标签。`/to-prd` 发 issue 时调 `gh label list` 发现缺，会 **自己** 把缺的 `gh label create` 补上，再发 issue —— 不用你一个个手动建。这就是它检测到「协议和现实有 gap」之后主动补齐。

## 为什么它不面试你

这是这一章的核心心智：

> **`/to-prd` 是一次机械综合，不是一次创造。**

它没有再做任何新设计 —— 所有设计在上一章的 grill 里 **已经做完了**。`/to-prd` 只是把那些决策塞进 PRD 模板。所以这一步又快又不容易出错。

反过来说：**如果你跳过 grill 直接 `/to-prd`，会得到一份「听着合理、每条都泛泛」的 PRD** —— 因为决策从没具体下来过。PRD 空洞不是 `/to-prd` 的锅，是 grill 没做够的锅。

这也解释了为什么它不该面试你：面试是 grill 的活。如果 `/to-prd` 又开始问你问题，说明这些问题本该在上一章问完。两个 skill 分工清楚 —— 一个负责把决策逼出来，一个负责把决策装起来。

## 小结

- `/to-prd` **不面试你**，直接把对话上下文 + `CONTEXT.md` + ADR 综合成一份 PRD。
- 唯一会停下来确认的是 **seam**（在哪测）：优先复用、取最高、越少越好。
- PRD 是固定七节模板，发成带 `ready-for-agent` 的 GitHub issue；发的时候会自己补齐缺的 triage 标签。
- PRD 是 **机械综合，不是创造** —— 设计在 grill 里就做完了。grill 越深，PRD 越实；grill 敷衍，PRD 越空。
