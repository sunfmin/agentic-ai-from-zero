# 第 12 章 —— 把模糊想法逼成决策

_上次校对：2026-06，对齐 Claude Code v2.x_

上一章你建好了一个有协议的空仓库，心里有一句话想法（本书示例：「一个读书清单小应用」）。这一章把那句话 **挤成一串具体决策**，而且决策会被当场写进文档。用的 skill 是 `/grill-with-docs`。

它是作者最受欢迎的 skill，也可能是整套里最值钱的一个。但它干的事一句话就能说清：**逼问你，然后把答案记下来。**

## `/grill-with-docs` = 逼问 + 记录

这个 skill 是两个更小的 skill 拼起来的：

- **`/grilling`（逼问）** —— 像面试官一样，**一次问一个问题**，沿着「决策树」一根枝一根枝往下走，每个问题都先给你一个 **推荐答案**。一次问一堆会把人问懵，所以它一次只问一个，等你回应了再问下一个。能靠读代码回答的问题，它就不问你、自己去读。
- **`/domain-modeling`（领域建模）** —— 在逼问的过程中 **顺手建术语字典**：顶撞你用错的词、把模糊的词收紧成精确的词、用具体场景压测你的概念边界；每定下一个词，**当场** 写进 `CONTEXT.md`；遇到难逆转的决定，做成一份 ADR。

合起来：你被逼着把想法想清楚，而「想清楚的产物」自动沉淀成文档。**留痕是免费的副产品，不是事后补的作业。**

> 💡 套件里还有个 `/grill-me`，只逼问、不建文档，适合非代码的事（比如想清楚一篇文章的结构）。`/grill-with-docs` 就是 `/grill-me` 加上「建 `CONTEXT.md` 和 ADR」这一层。要做项目，用带 docs 的这个。

## 逼问长什么样

关键不在「它会问问题」，而在 **每个问题它都帮你把功课做了一大半**。它不会干巴巴丢给你「你想用 A 还是 B？」。每一题通常给你：

- **问题背景** —— 这件事为什么是个需要决定的决定；
- **三四个候选** —— 以及各自的代价；
- **一个明确的推荐** —— 外加几条「为什么推荐它」；
- **为什么不选别的** —— 防止你没想清楚就选。

于是你大多数时候看一眼就能在半分钟内判断「行 / 不行」，几乎不用现场苦思。真正有价值的，是那些你回答「**不，我想要 X 不是 Y**」的时刻 —— 那是 Claude 替你做不了的部分。

根目录的 [`BOOTSTRAP-CASE-STUDY.md`](../../BOOTSTRAP-CASE-STUDY.md) 完整复盘了一次真实 session：作者用这个 skill 把「我想做一本教程」逼成了 **14 个具体决策**（操作系统？终点是会写 skill 还是会对话？双语怎么做？读者是谁？……）。想看真实问答长什么样，去读它 —— 那本教程，就是你现在读的这本。

## 动手：逼问你的想法

回到上一章那个练手仓库（示例里是 `~/reading-list`），开 Claude。把你那句话想法跟在 `/grill-with-docs` 后面发出去。

> 🛠 试一试
>
> 在 prompt 上输入（把想法换成你自己的）：
>
> > `/grill-with-docs 我想做一个小应用，记录我想读的书和文章，能加条目、打标签、标记读完，看看还有多少没读。`
>
> 你应该看到：Claude **不会** 马上开始写东西。它会先抛出 **第一个** 问题 —— 通常是最根上、最影响后面的那个（比如「这个清单是只给你自己用，还是要多人协作？这决定了要不要账号系统」），并给出它的推荐和理由。然后停下，等你。

接下来就是你和它来回走决策树。每一题，你可以：

- 觉得推荐合理 → 答「按推荐」或「同意」；
- 有自己的想法 → 答「不，我要 X」，并说一句为什么；
- 想反过来问它 → 直接问，它会答完再回到正题。

> 🛠 试一试
>
> 一路答下去，直到它问完（小项目通常五到十几个问题）。中途留意两种时刻：
>
> - 它 **顶你的词**：比如你一会儿说「文章」一会儿说「条目」，它会问「这两个是一回事吗？我们统一叫哪个？」——这是 domain-modeling 在收紧术语。
> - 它 **能查就不问**：如果某个问题翻一眼代码就有答案，它会去读代码，而不是问你。（这个空仓库里还没代码，所以这一章你基本看不到；第 15 章对着有代码的仓库就明显了。）

## 留意「副产品」当场长出来

逼问到一半，不用你提醒，Claude 会 **自己** 建文件。这是这个 skill 最值钱的地方。

> 🛠 试一试
>
> 问到几个术语定下来之后，开另一个终端窗口（或者让 Claude 列一下），看看：
>
> 输入：`cat CONTEXT.md`
>
> 你应该看到：一个术语字典慢慢成形 —— 你和 Claude 刚统一的那些词（「条目」「标签」「读完」是什么意思、彼此什么关系），一条条写在里面。它 **只放术语**，不放实现细节，也不当草稿本。

ADR 则建得 **很克制**。Claude 只在一个决定 **三条同时成立** 时才提议建 ADR：

1. **难逆转** —— 以后改主意代价不小；
2. **不解释会让人困惑** —— 未来读代码的人会问「为什么当初这么搞」；
3. **是真权衡的结果** —— 当时真有别的选项，你为某些理由选了这个。

三条缺一条，它就不建。所以你不会被一堆鸡毛蒜皮的 ADR 淹没。

> 🛠 试一试
>
> 如果逼问中真出现了这种决定（比如「读完的条目是软删除还是真删」），Claude 会问你 **要不要** 记成 ADR。答要，然后看：
>
> 输入：`ls docs/adr/`
>
> 你应该看到：一个 `NNNN-标题.md` 文件，里面写清了这个决定、当时的几个选项、以及为什么选了现在这个。半年后你自己回来都能看懂。

## 逼问结束 = 决策定稿

问完之后，你手上有的不只是「聊明白了」，而是：

- 一个仓库，业务代码还是一行没有；
- 但所有 **可能引起后期返工的争议**（要不要账号？多端同步？删除语义？）都已经有答案；
- 而且答案 **有据可查** —— `CONTEXT.md` 是术语字典，`docs/adr/` 是难逆转决定的理由书。

这一步把「模糊想法」收成了「定稿的决策」。**真正的设计工作，在这一章就做完了。** 下一章的 `/to-spec` 不会再做任何新设计 —— 它只是把这些决策装进模板。所以，逼问越认真，后面越省事；逼问敷衍，后面的 PRD 就会泛泛而空。

把这一步的产物存个档：

> 🛠 试一试
>
> 对 Claude 说：`把 CONTEXT.md 和 docs/ 下的东西 commit 一下并 push。`
>
> 你应该看到：Claude 把 `CONTEXT.md` 和那些 ADR 一起 commit、推上 GitHub。

## 小结

- `/grill-with-docs` = `/grilling`（逼问）+ `/domain-modeling`（建术语字典和 ADR）。
- 逼问 **一次只问一个问题**，每题都给背景、候选、推荐、和「为什么不选别的」。
- 最值钱的是你答「不，我要 X」的时刻 —— 以及 **当场长出来的** `CONTEXT.md` 和 ADR。
- ADR 建得很克制：**难逆转 + 不解释会困惑 + 真有权衡**，三条都满足才建。
- 真正的设计在这一步做完；后面 PRD 和 issue 只是机械综合。逼问的深度，决定后面一切的质量。
