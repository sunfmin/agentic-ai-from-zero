# 第 6 章 —— 写你的第一个 skill

_上次校对：2026-05，对齐 Claude Code v2.x_

你将手写一个 `SKILL.md`、装好它、看着它在你主项目的草稿上被触发。读完这一章，你在第 4 章拷进 `~/writing` 的那三篇草稿就会被自动审稿 —— 每次你提到想审稿，它就会出现。

## skill 住在哪儿

skill 住在 `~/.claude/skills/` 下面。每一个 skill 是一个子文件夹。文件夹的名字本身不参与触发匹配 —— 真正参与的是 `SKILL.md` 里 `name` 字段 —— 但按惯例文件夹用 skill 的名字命名。

> 🛠 试一试
>
> 输入：`mkdir -p ~/.claude/skills/review-writing`
>
> 然后：`ls ~/.claude/skills/`
>
> 你应该看到：列表里有 `review-writing`（以及之前你已有的其他 skill）。

## 写这个 SKILL.md

用你熟悉的编辑器打开这个新文件夹（或者 `open ~/.claude/skills/review-writing`，在 Finder 窗口里新建文件）。建一个叫 `SKILL.md` 的文件，内容如下。

把它打进去（或者贴进去）。措辞本身没那么关键，关键是**形态** —— 顶上 frontmatter、下面 body。

```markdown
---
name: review-writing
description: |
  审阅草稿（markdown 文件里的散文、公告、会议总结、上线笔记、
  内部沟通文等等），检查四类常见写作问题：啰嗦、被动语态、用抽象
  名词代替本可以用的具体动词、套话。当 user 要求 "审"、"改"、
  "批评"、"改进"、"压缩" 或 "给反馈" 一篇草稿，或者贴出 markdown
  文件问哪儿不好，就触发。
---

# 审阅写作

当 user 要求审稿时，按以下步骤：

1. 读 user 指的那个文件。如果他们没说哪个，问一下。
2. 用下面四条**原则**检查文章。每发现一个问题，引出原句，提出一个更紧凑的改写建议。
3. 最后给一句整体评价，说出这篇草稿在四条原则里最严重的是哪一条。

## 原则

**P1 —— 一句话讲一件事。** 用"而且"、"，这"、分号把两个或更多独立分句串起来，通常背后是结构问题。拆开。一句话超过约 50 字就要警觉是不是 P1。

**P2 —— 主动语态。** "X 被 Y 完成"几乎永远比"Y 做了 X"读起来差。寻找"被 + 动词"结构，改成主动。

**P3 —— 用具体动词，不用抽象名词。** "对它进行了改进" → "改进了它"。"工具的部署被完成了" → "部署了工具"。把动词名词化（改进、部署、完成、识别、优化）的句子，多半 P3 中招。

**P4 —— 砍掉套话和废话。** 那些听上去重要、其实啥也没说的短语：赋能、抓手、闭环、向前看、归根结底、价值创造、转型旅程、卓越运营。删掉它们，句子通常还成立。

## 输出格式

每一条问题用这个形态：

> **[原则编号]** 原句："..."
> 改写："..."

最后：

> **整体：** 这篇草稿最严重的问题是 [原则编号]。[一句话总结。]

具体、直接。不要用"也许可以考虑"、"你或许想……"软化。user 要的是审稿，给一份审稿。
```

> 🛠 试一试
>
> 存盘之后，输入：`cat ~/.claude/skills/review-writing/SKILL.md | head -10`
>
> 你应该看到：你刚写进去的那个文件前 10 行 —— frontmatter 块加上 body 的开头。

## 看着它触发

回到你的主项目 —— 第 4 章那个写作文件夹。

> 🛠 试一试
>
> 输入：`cd ~/writing && claude`
>
> 你应该看到：Claude 的常规欢迎和 prompt。

这个 skill 现在对 Claude 是可见的。你不需要"加载"任何东西。Claude 会自动扫 `~/.claude/skills/`。

触发它：

> 🛠 试一试
>
> 在 Claude 的 prompt 上输入：
>
> > 帮我审一下 `announcement.md`。
>
> 你应该看到：Claude 读 `announcement.md`，然后给一份审稿。看输出里有没有 **P1**、**P2**、**P3**、**P4** 这些编号 —— 这些是你 skill body 里的原则编号。看到编号 = skill 触发了。看到的是一份没有编号的泛泛而谈的审稿 = skill 没触发，去看下面的"排错"。

试一下另外两篇：

> 🛠 试一试
>
> > 批评一下 `launch-post.md`。
>
> 然后：
>
> > 给我 `standup-summary.md` 的反馈。
>
> 你应该看到：每一篇都得到一份同样形态的审稿 —— 引出原句、附上原则编号、最后一句整体评价。

"审一下"、"批评"、"给反馈"三种说法都能触发同一个 skill —— 因为 description 里的触发关键词把这三种说法都明确列出来了。

## 排错 —— skill 没触发

如果你拿到了一份**没有原则编号**的审稿，那 skill 没起来。最可能的原因是 description。检查：

- `SKILL.md` 是不是确实在 `~/.claude/skills/review-writing/SKILL.md`？不是 `Skill.md`、不是 `skill.md`、不是 `~/skills/SKILL.md`。
- 文件存盘了吗？（在新 terminal 里 `cat ~/.claude/skills/review-writing/SKILL.md` 看一下。）
- 建好文件之后有没有开一个**新的** `claude` session？老 session 不会感知到新加的 skill。
- frontmatter 的分隔线是不是确实是单独成行的三个 dash？上下两行都要。

如果结构都对、但像"帮我审一下这篇草稿"这种说法还是不触发，问题很可能在 description 的措辞。第 8 章专门讲怎么调 description；到时候再回来调它。

## 小结

- 硬盘上的一个 skill = `~/.claude/skills/` 下面的一个文件夹，里面有 `SKILL.md`。
- 不需要"重新加载"、不需要注册、不需要任何命令 —— `claude` 启动 session 时自动发现。
- 触发与否取决于 description 的措辞跟你说的话匹不匹配。
